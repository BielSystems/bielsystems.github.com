  • Desenho das telas
    O cliente e o time desenham como imaginam o projeto que será desenvolvido. Saiba mais!

    Desenho das telas

    Segunda-feira

    O objetivo desta etapa é extrair as principais ideias de todos os envolvidos e melhorar a comunicação entre o cliente e os membros da equipe em relação ao entendimento do que precisa ser feito.

    Quando falamos desenho das telas, quer dizer literalmente um desenho, simples, com papel e caneta. Veja um exemplo!

    1. Todos do time e o cliente desenham individualmente as telas de como que imaginam que será a interface;

    2. Os desenhos devem refletir os fluxos das telas da aplicação. Como deve ser a interação do usuário com a tela, quais elementos devem existir e tudo mais;

    3. Todos apresentam seus desenhos e há a discussão sobre as melhores soluções;

    4. Cria-se o desenho final que será o documento de entrada para a próxima etapa.

  • Quebra de tarefas e estimativa
    O time quebra o desenho final em tarefas de design e programação e atribui uma estimativa individual. Saiba mais!

    Quebra de tarefas e estimativa

    Segunda-feira

    O principal objetivo aqui é saber qual o esforço para se desenvolver uma funcionalidade, quanto de esforço será necessário para desenvolver a mesma.

    1. O time discute as telas desenhadas anteriormente e cria histórias baseada em cada uma das telas;

    2. O time desenha no quadro os componentes necessários para criar cada uma das telas;

    3. Todos jogam a pontuação de cada uma das tarefas, designers e desenvolvedores juntos;

    4. A pontuação é sempre 1, 2, 4 ou 8 que representam o esforço do time em relação a tal tarefa;

  • Priorização das tarefas
    O cliente prioriza as tarefas que serão realizadas durante a semana. Saiba mais!

    Priorização das tarefas

    Segunda-feira

    Depois de se saber o esforço de cada uma das tarefas é hora de priorizar, definindo quais das tarefas são mais importantes para serem desenvolvidas durante a semana.

    1. O cliente coloca em ordem de prioridade todas as tarefas, independente de pontos;

    2. O time ajuda o cliente a priorizar, mostrando a complexidade de cada uma das tarefas;

    3. O cliente prioriza em média 30 pontos por pessoa no time. Ou seja um time com 2 pessoas entrega aproximadamente 60 pontos;

    4. Ao final da priorização o cliente e o time já sabem o que será desenvolvido durante a semana;

    Ambiente de Testes

    Terça e Quarta-feira

    Todo projeto possui um ambiente de testes, chamado de staging, funcionando online para o cliente e o time realizarem testes em um ambiente mais próximo do de produção.

    1. O ambiente de staging é protegido com login e senha para que só o pessoal autorizado tenha acesso ao mesmo;

  • Boas práticas
    O time utiliza de testes automatizados e todas as demais práticas do XP. Saiba mais!

    Boas práticas

    Terça e Quarta-feira

    O time de desenvolvimento utiliza diversas boas práticas entre elas testes automatizados, 100% de cobertura de testes e integração continua.

    1. Todo código que é escrito antes é escrito um teste para o mesmo garantindo o seu funcionamento, assim novas funcionalidades não quebram funcionalidades antigas;

    2. Não adianta testar se não testamos tudo! Então garantimos que temos 100% de cobertura de testes, ou seja não tem uma linha de código que não foi testada;

    3. Toda pequena porção de código desenvolvido é enviado para o ambiente de staging, ocorrendo diversas vezes por dia este processo, garantindo que não haja nenhuma surpresa no ambiente de produção.

    Roteiro de testes

    Quinta-feira

    Com a finalização do desenvolvimento na quinta-feira o cliente é notificado sobre isso e recebe um roteiro de testes para testar todo o sistema.

    1. O roteiro de testes é um passo a passo de como testar cada uma das funcionalidades;

    2. O cliente anota qualquer observação sobre a entrega feita diretamente no roteiro de testes;

  • Testando o sistema

    Testando o sistema

    Quinta-feira

    1. O cliente tem de quinta-feira a noite até o domingo para testar todo o sistema;
