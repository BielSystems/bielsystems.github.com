Segunda-feira

  • Ideia do cliente
    No primeiro encontro, cliente e time conversam sobre o projeto e suas visões de curto e longo prazo.
  • Desenho das telas
    Cliente e time desenham as páginas como imaginam que o projeto será desenvolvido. Saiba mais!

    Desenho das telas

    Segunda-feira

    O objetivo desta etapa é extrair as principais ideias de todos (cliente e time) e através da comunicação, chegar ao acordo do que precisa ser feito.

    Quando falamos desenho das telas, queremos dizer que é feito, literalmente, um desenho do que você imagina como será a sua página. Simples, com papel e caneta. Veja um exemplo!

    1. Todos do time e o cliente desenham, individualmente, as interfaces das telas imaginadas;

    2. Os desenhos devem refletir os fluxos das telas da aplicação. Como deve ser a interação do usuário com ela, quais elementos devem existir e tudo mais;

    3. Todos apresentam seus desenhos e há a discussão sobre as melhores telas;

    4. Cria-se o desenho final, que será o documento de entrada para a próxima etapa.

  • Quebra de tarefas e estimativa
    O time dissolve o desenho final das telas em tarefas de design e programação, atribuindo uma estimativa individual. Saiba mais!

    Quebra de tarefas e estimativa

    Segunda-feira

    O principal objetivo aqui é saber qual e quanto esforço será necessário para o seu desenvolvimento.

    1. Após o time discutir as telas desenhadas, cria-se um passo-a-passo dos detalhes de cada tela. Chamamos este de "histórias", onde dentro de cada uma, há várias tarefas;

    2. Na sala onde o time se encontra, é desenhado no quadro-branco os componentes necessários para criar cada uma das telas;

    3. Cada tarefa tem um valor x de pontos. Todos do time estipulam as pontuações de cada uma das tarefas;

    4. A pontuação é sempre 1, 2, 4 ou 8, que representam o esforço do time em relação a tal tarefa;

  • Priorização das tarefas
    O cliente prioriza as tarefas que serão realizadas durante a semana. Saiba mais!

    Priorização das tarefas

    Segunda-feira

    Depois de se saber o valor de esforço de cada uma das tarefas, é hora de priorizá-las, definindo quais delas são mais importantes para serem desenvolvidas durante a semana.

    1. O cliente coloca em ordem de prioridade todas as tarefas, independentemente dos pontos;

    2. O time ajuda o cliente a priorizar, mostrando a complexidade de cada uma das tarefas;

    3. O cliente prioriza em média 30 pontos por pessoa no time. Ou seja, um time com 2 pessoas, entrega, aproximadamente, 60 pontos;

    4. Ao final da priorização, o cliente e o time já sabem o que será desenvolvido durante a semana;

Terça e Quarta-feira

  • Desenvolvimento
    O time trabalha nas tarefas priorizadas pelo cliente. Saiba mais!

    Desenvolvimento

    Terça e Quarta-feira

    No desenvolvimento, o time começa a fazer as tarefas que foram definidas na segunda-feira.

    1. O time sempre está em comunicação entre si, assim todos sabem como está o andamento do projeto;

    2. O código do projeto é alterado por todos do time. Assim, todos possuem conhecimento completo do sistema;

    3. Não se preocupe! Seu time estará se divertindo desenvolvendo seu projeto enquanto comem biscoitinhos e bebem Red Bull e Heineken;

  • Ambiente de Testes
    O projeto que está em desenvolvimento fica disponível para o cliente em um ambiente de testes online. Saiba mais!

    Ambiente de Testes

    Terça e Quarta-feira

    Todo projeto possui um ambiente de testes, chamado de staging. Ele funciona online para que o cliente e o time realizem testes em um ambiente mais próximo do que está em produção.

    1. O ambiente de staging é protegido com login e senha para que somente as pessoas autorizadas tenham acesso;

  • Boas práticas
    O time utiliza testes automatizados e todas as demais práticas do XP (Xtreme Programming - Metodologia ágil). Saiba mais!

    Boas práticas

    Terça e Quarta-feira

    O time de desenvolvimento utiliza diversas boas práticas, dentre elas: testes automatizados, 100% de cobertura e integração contínua.

    1. Todo código escrito já foi um teste para ter garantia de seu funcionamento. Assim, novas funcionalidades não afetam as antigas;

    2. Temos que testar tudo! Após isto, garantimos 100% a cobertura de seus testes. Ou seja, não há uma linha de código sequer que não tenha sido testada;

    3. Toda pequena porção de código desenvolvido é enviado para o ambiente de staging, ocorrendo diversas vezes por dia este processo, garantindo que não haja nenhuma surpresa no ambiente de produção.

Quinta-feira

  • Finalização do desenvolvimento
    O time finaliza as tarefas da iteração. Saiba mais!

    Finalização do desenvolvimento

    Quinta-feira

    Na quinta-feira, todas as tarefas priorizadas na segunda-feira são finalizadas e estarão disponíveis no ambiente de staging.

  • Roteiro de testes
    No final do dia, é entregue o roteiro de testes para o cliente. Saiba mais!

    Roteiro de testes

    Quinta-feira

    Com a finalização do desenvolvimento na quinta-feira, o cliente é notificado e recebe um roteiro de testes para testar todo o sistema.

    1. O roteiro de testes é um passo-a-passo de como testar cada uma das funcionalidades, feito diretamente no ambiente de testes;

    2. O cliente anota qualquer observação sobre a entrega feita diretamente no roteiro de testes;

  • Testando o sistema
    O cliente tem de quinta-feira à noite até segunda-feira da semana seguinte para testar o que foi desenvolvido e trazer suas observações para a próxima reunião.
